Output is JSON by default. Add `--pretty` for human-readable output.

## Commands

| Command | What it does |
|---------|--------------|
| `hibi init` | Create `.claims/` (with a per-repo banner nonce). |
| `hibi record` | Record a span-first claim: a doc span, anchored to zero or more code spans. |
| `hibi check` | Verify every claim. Add `--write` to stamp status banners into affected docs. |
| `hibi diff --since <ref>` | Show what a change invalidated, relative to a base ref. |
| `hibi status --doc <doc>` | Report whether a doc is still current. |
| `hibi query --path <path>` | Before editing: list which claims cover a file. |
| `hibi suggest --doc <doc>` | Propose anchorable claims from a document (creates `suggested` records). |
| `hibi reanchor <claim-id>` | Re-resolve a drifted claim against current doc/code content. |
| `hibi supersede --new <doc> --old <doc> --type <type>` | Record that one doc supersedes/amends another. |

### `hibi record`

Span-first: the documented sentence's span supplies the claim text (the doc
side), and you anchor zero or more code spans that back it. `--text` survives
only as a legacy override for the rare pristine case where no doc span applies.

```sh
hibi record \
  --doc README.md --doc-quote "Retries are capped at 5 attempts" \
  --code-file src/retry.ts --code-quote "MAX_ATTEMPTS = 5" \
  --trust verified --owner alice
```

| Flag | Description |
|------|-------------|
| `--doc` | The document the claim lives in. |
| `--doc-quote` / `--doc-range L42:L44` / `--doc-line <n>` | The doc span that supplies the claim text. |
| `--code-file` | The source file the claim is anchored to. |
| `--code-quote` / `--code-range L1:L9` / `--code-line <n>` | The code span to anchor on (matched fuzzily). |
| `--coarse` / `--glob <g>` | Anchor to a whole file or glob (navigation / blast-radius only). |
| `--trust` | Authored trust: `verified` \| `inferred` \| `assumed`. |
| `--enforce` / `--enforcement <e>` | Mark `enforced` (gates) — or set `suggested` \| `enforced` \| `retired` \| `unanchored-legacy`. |
| `--claim-kind <k>` | Declare a behavioral kind (e.g. `ordering`, `retry`, `concurrency`). |
| `--verifier kind:ref` | Link an executable verifier (repeatable). |
| `--ttl <iso>` | Expire the claim after an ISO-8601 instant. |
| `--owner` | Who vouches for the claim. |
| `--text` | Legacy override: the claim sentence when no doc span applies. |

## The verdict model

Hibi never collapses a claim into one word. A verdict is **two axes plus a
flag** (no `fresh`/`stale`/`ghost` enum):

- **Anchor resolution**, reported per side as `doc:…` / `code:…`:
  `unchanged` · `moved` · `changed` · `ambiguous` · `orphaned`.
- **Behavioral belief** (only on behavioral claims):
  `unverified` · `at-risk` · `supported` · `refuted`.
- **`expired`** — an orthogonal TTL flag, not a state.

A verdict reads, e.g., `doc:unchanged · code:changed · behavior:at-risk`. The
JSON is verdict-first — `{ doc, code, behavior?, expired, gates, evidence{…} }`
— so a truncated read still surfaces the decision. The words "drift" and "stale"
are only the human banner roll-up, never machine states.

## Exit codes

Hibi communicates results through its exit code so it slots straight into CI and
git hooks:

| Code | Meaning |
|------|---------|
| `0` | All clean. |
| `2` | Gating: `changed` / `orphaned` / `ambiguous` / `expired` / `refuted` on an **enforced** claim. |
| `3` | `moved` / `at-risk` (re-anchorable warning). |
| `1` | Operational error. |

## Strictness

Tune which states fail the run with `--fail-on`:

```sh
hibi check --fail-on gating    # default — fail on gating verdicts
hibi check --fail-on warn      # also fail on moved / at-risk warnings
hibi check --fail-on tamper    # fail only on tampering
hibi check --fail-on never     # report, never fail the run
```

<Tip>
  In CI, pair `hibi check` with `--fail-on` to choose how strict the gate is.
  The official GitHub Action wraps exactly this.
</Tip>
