What changed in the spec —
from X to Y, and why

hibi is a tool that catches documentation that no longer matches the code. We re-opened two big design decisions, ran deep research, and rewrote the spec. Here is every change in plain words — what it was, what it is now, and the reason.

1 · Now watches both sides (docs and code) 2 · The document is the source of truth, not a copy 3 · Regex advisor → change-gated risk routing 4 · No AI model decides a verdict (here's why) 5 · New two-axis state model (borrowed words, not invented) 6 · New commands: span-first record, suggest, reanchor

In plain English — what hibi does

You write a sentence in a doc like "Retries are capped at 5 attempts." hibi remembers where that sentence lives and which code it talks about. Later, when something moves, hibi re-checks the link and — if it looks out of date — stamps a visible warning banner right into the doc, so a human (or an AI coding agent) never trusts a stale page.

The golden rule that never changed: hibi is deterministic. Same inputs → same answer, every time. That trust is the whole product. Every change below protects it.

Notice: the old check only ever looked at code. The new check looks at the document first, then the code, then behavior.

1

It now catches drift on both sides

Before (X)

hibi only noticed when the code changed. If someone edited or deleted the doc sentence itself, hibi never noticed — it kept checking a sentence the doc no longer said.

After (Y)

A claim now binds a document span ↔ code. hibi watches both ends: code drift and doc drift (the sentence was reworded, moved, or removed).

flowchart LR
  D["`Doc sentence
  'Retries capped at 5'`"] <-->|claim| C["`Code
  MAX_ATTEMPTS = 5`"]
  D -.->|"edited / deleted?"| W1["`doc drift caught`"]
  C -.->|"changed / moved?"| W2["`code drift caught`"]

A claim is a two-way link. Break either end and hibi flags it.

Why

The whole point is "no drift between docs and code." Watching only one side left a silent hole: an agent could rewrite a sentence into something never verified, and hibi would say everything's fine.

2

The document is the source of truth — not a copy

This is the deepest change. It's why change #1 was even possible.

Before (X) — two truths

hibi copied the sentence into its side store (.claims/) and only pointed at the code. Now there were two copies of the claim — the doc's, and the store's — and they could quietly disagree.

After (Y) — one truth

The store keeps pointers (anchors), not the prose. The real claim text is read live from the document at check time. One source of truth: the file the agent actually reads.

flowchart TB
  subgraph B["BEFORE — store owns a copy"]
    direction LR
    BD["`README says:
    'capped at 10'`"]
    BS["`store copy:
    'capped at 5'`"]
    BD -. can disagree .- BS
    BS --> BV["`verifies the
    STALE copy`"]
  end
  subgraph A["AFTER — store owns a pointer"]
    direction LR
    AD["`README says:
    'capped at 10'`"]
    AS["`store: anchor →
    points into README`"]
    AS --> AD
    AD --> AV["`verifies the
    LIVE sentence`"]
  end

"Model C+": the artifact is authoritative; the store just points at it (with a tiny optional ID for owned docs, and a legacy copy mode only for read-only files).

Why

A copy that outlives the sentence it copied is a landmine. The thing an agent reads is the current document — so that must be what hibi trusts and verifies.

Evidence

This mirrors how web annotation works (W3C / hypothes.is): records point into a page. A study of 20,953 annotations found ~22% "orphaned" when pages changed — proof that a deleted target must be an explicit state, not silent "all good."

3

The regex "semantic advisor" → change-gated risk routing

Before (X) — noise machine

The old "semantic" tier matched keywords in the sentence ("retry", "sort", "cache"…) and warned every single run, forever — even when nothing changed. It looked at the wording, never at whether anything actually moved.

After (Y) — quiet until it matters

Behavioral Risk Routing: a behavioral claim is flagged only when related code actually changed (the function, its callees, its config, its literals). And you can attach a real test that turns "risk" into a true pass/fail.

flowchart TD
  S["`Claim describes behavior?
  (e.g. 'retries with backoff')`"] -->|no| Q["`stay silent (n/a)`"]
  S -->|yes| G{"`Did related code change?
  (callees / imports / config)`"}
  G -->|no| Q2["`unverified
  (resting)`"]
  G -->|yes| V{"`Is a verifier linked?`"}
  V -->|no| R["`at-risk
  (advisory: re-verify)`"]
  V -->|verifier passes| OK["`supported`"]
  V -->|verifier fails| F["`refuted
  (can gate)`"]

The "gate" is the word did related code change? — that's what kills the noise.

Why

A warning that fires constantly gets ignored — and then people stop trusting all of hibi's output, including the rock-solid parts. Noisy analyzers report up to 96% useless warnings. A warning tied to a real change is a warning people act on.

4

No AI model is allowed to decide a verdict

We genuinely re-opened this ("could an LLM judge if a claim is still true?"). The research answered: not reliably enough to trust. So determinism stays — now backed by evidence, not just belief.

F1 0.58

Best published AI doc-vs-behavior checker (METAMON). It misses ~half of real problems.

63%

Wrong answers an LLM "judge" still accepted as correct in one audit.

~0.57

Judge consistency under normal sampling — same input, different answers.

The rule we landed on: "truth needs an oracle"

Only running something (a test, a contract, a snapshot) can prove behavior is still true. A model can guess. So we split the job by who's good at what:

hibi engine
points at where to look (deterministic).
Tests
certify truth when they exist.
The agent
already in the loop — it judges the gray areas.

An LLM may still help — but only as an opt-in advisor that explains or triages. It can never set, clear, or override a verdict.

5

The state words got a complete revamp: a two-axis model

This is the star of this revision. The old verdict was three flat lists of words picked from three unrelated metaphors. The new verdict is two clean axes, each with words borrowed from a field that already solved the same problem.

Two questions → two entities → two vocabularies

There are really only two questions hibi answers about a claim — and the old model muddled them together. Separate them and the whole vocabulary falls out:

Axis 1 · Anchor resolution

"Can I still find the span — and is it unchanged?"

A location question. One vocabulary, applied to each side (reported as doc:… and code:…).

  • unchanged — found, identical
  • moved — found, relocated
  • changed — found, content differs
  • ambiguous — matches in 2+ places
  • orphaned — span deleted / unresolvable
Axis 2 · Behavioral belief

"Do we still believe the documented behavior holds?"

A truth question. Absent on non-behavioral claims (shown as n/a).

  • unverified — behavioral, untested, resting
  • at-risk — reachable code changed; belief no longer justified
  • supported — a linked verifier passed
  • refuted — a linked verifier failed (the only one that can gate)

Plus one orthogonal flag, expired (the TTL elapsed) — deliberately not a state, because "is it past its time?" is a different axis from "where is it / is it true?".

A verdict now reads: doc:unchanged · code:changed · behavior:at-risk

Notice: both sides of Axis 1 share the same five words; Axis 2 is separate; expired hangs off to the side as a flag, not a state.

Where each word came from (nothing invented)

Each word is borrowed from an established source, so it's already familiar to anyone who's met the source field. Only one had to be coined.

unchanged · moved
← git status — the most widely-understood "resolve a tracked thing vs current state" vocabulary.
ambiguous
← W3C Web Annotation — its word for a quote that matches in multiple places.
orphaned
← hypothes.is — its term for an annotation that can no longer anchor; now on both sides.
supported · refuted
← FEVER, the fact-/claim-verification standard. Our unit literally is a claim.
unverified
← the SMT-solver / Frama-C / Nagios "unknown / not-established" family.
at-risk
← reason-maintenance / JTMS: "a belief loses its justification when its support changes." The one word no off-the-shelf vocabulary named.
Why these exact words? (the ubiquitous-language insight)

The old model had three flat enums named by three unrelated metaphors: food (fresh/stale), spooky (ghost), and literal (doc-edited). That hid two problems. (1) The same concept appeared twice under different words — code ghost ≈ doc doc-orphaned, and code stale ≈ doc doc-edited. (2) Three metaphors for what are really only two questions.

The fix: realise there are exactly two questions → two entities → two vocabularies, each borrowed from a field that already named the idea, so the words are familiar on sight. And changed was chosen over stale for the leaf state on purpose: unchanged/changed now form a clean antonym pair, and the word "drift / stale" stays free to be the human roll-up in a banner headline — instead of being overloaded as a machine state.

Separately — a "switch": is this claim actually enforced yet?

This enforcement lifecycle is its own thing, untouched by the revamp and orthogonal to the two axes above.

stateDiagram-v2
  [*] --> suggested: hibi proposes it
  suggested --> enforced: confirmed (agent / human)
  enforced --> retired: withdrawn
  suggested --> retired: dismissed
  note right of suggested
    advisory only —
    never blocks a build
  end note
  note right of enforced
    can stamp banners
    and fail CI
  end note

Why a switch? Auto-guessed claims are too noisy to trust. A claim only gets teeth (enforced) after someone confirms it.

6

New & changed commands

Before (X) — text-first record
hibi record \
  --text "Retries capped at 5" \
  --file src/retry.ts \
  --quote "MAX_ATTEMPTS = 5"

You typed the claim text in — so the store got a copy that could drift from the doc.

After (Y) — span-first record
hibi record \
  --doc README.md --doc-range L42:L44 \
  --file src/retry.ts \
  --code-quote "MAX_ATTEMPTS = 5"

You point at the sentence in the doc; hibi reads the text from the file. The doc stays the source of truth.

hibi suggest

Scans a new/changed doc and proposes candidate claims (anchorable, verifiable ones — skips opinions and background). They start as suggested until confirmed.

hibi reanchor <id>

Re-points a drifted claim at the current doc and code, refreshes the anchors, and resets its state to unchanged. For an orphaned claim, asks for the new location (or retire it).

Glossary

claim — a sentence in a doc, linked to the code it describes anchor — a robust "pointer" to a span of text/code that survives edits bidirectional — pointing at both the doc side and the code side source of truth — the one authoritative copy (now: the live document) deterministic — same input → same output, always; no randomness change-gated — only fires when something actually changed verifier — a real test/check that proves behavior (the "oracle") orphaned — the anchored sentence is gone; a real signal, not "fine" anchor resolution — Axis 1: "can I still find the span, and is it unchanged?" behavioral belief — Axis 2: "do we still believe the documented behavior holds?" ubiquitous language — one shared, precise vocabulary; words borrowed, not invented LLM-as-judge — using a model to grade correctness (rejected for verdicts) banner — the visible status hibi stamps into the doc
